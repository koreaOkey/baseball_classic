## ADDED Requirements

### Requirement: 사용자 신고
시스템은 사용자가 메시지를 신고할 수 있는 기능을 제공해야 한다(SHALL). 신고 사유는 최소 5가지(욕설·비방, 혐오·차별, 도배·스팸, 개인정보 노출, 기타)를 포함해야 한다.

#### Scenario: 메시지 신고
- **WHEN** 사용자가 메시지를 길게 눌러 신고를 선택하고 사유를 선택해 제출한다
- **THEN** 시스템은 신고를 접수하고 "24시간 내 검토" 안내 토스트를 표시한다

#### Scenario: 신고 누적 자동 블라인드
- **WHEN** 동일 메시지에 대해 서로 다른 사용자로부터 신고가 3회 누적된다
- **THEN** 시스템은 즉시 해당 메시지를 자동 블라인드 처리한다

### Requirement: 금칙어 필터
시스템은 메시지 작성 시 금칙어 DB에 등록된 표현을 탐지해 자동 블라인드 처리해야 한다(SHALL). 금칙어는 선수 실명과 욕설 조합을 포함해 운영 중에도 추가 가능해야 한다.

#### Scenario: 금칙어 포함 메시지
- **WHEN** 사용자가 금칙어가 포함된 메시지를 전송한다
- **THEN** 시스템은 메시지를 전송은 허용하되 즉시 블라인드 상태로 저장하고, 다른 사용자에게는 "신고된 메시지" 플레이스홀더로 표시한다

### Requirement: AI 모더레이션
시스템은 메시지 저장 시 AI 모더레이션 서비스를 호출해 부적절 점수를 평가하고(SHALL), 임계치를 초과한 메시지는 자동 블라인드해야 한다(SHALL).

#### Scenario: AI 임계치 초과
- **WHEN** 사용자가 메시지를 전송하고 AI 모더레이션 점수가 임계치를 초과한다
- **THEN** 시스템은 해당 메시지를 자동 블라인드하고 작성자에게만 원문을 보여준다

### Requirement: 24시간 검토 SLA
시스템은 사용자가 접수한 신고에 대해 24시간 이내에 운영자 검토 결과(유지·복구·제재)를 반영해야 한다(SHALL).

#### Scenario: 신고 검토 및 유지
- **WHEN** 운영자가 24시간 이내에 신고된 메시지를 검토해 부적절 판단을 내린다
- **THEN** 시스템은 해당 메시지를 영구 블라인드하고 작성자에게 경고를 기록한다

#### Scenario: 신고 검토 후 복구
- **WHEN** 운영자가 신고된 메시지를 검토해 정상 판단을 내린다
- **THEN** 시스템은 자동 블라인드를 해제하고 메시지를 다시 표시한다

### Requirement: 내부 식별자 보관
시스템은 익명으로 작성된 메시지에 대해서도 실제 user_id를 내부에 보관해야 한다(MUST). 이는 법적 요청 대응을 위한 최소 보관 의무를 충족한다.

#### Scenario: 익명 메시지 내부 식별자
- **WHEN** 메시지가 익명 플래그로 저장된다
- **THEN** 시스템은 공개 표시용 정보(닉네임·프로필)는 가리되 데이터베이스에는 작성자 user_id를 함께 저장한다
