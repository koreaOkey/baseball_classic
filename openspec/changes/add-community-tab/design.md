## Context

야구봄은 실시간 중계·햅틱 피드백에 집중된 앱으로, 현재 사용자 간 상호작용이 거의 없다. 기존 앱은 로그인 없이 동작하지만, 커뮤니티 탭을 도입하면서 소셜 로그인, 사용자 프로필, 실시간 채팅, 기사 수집, 모더레이션 파이프라인까지 한꺼번에 신규 도입해야 한다. 법적으로는 네이버·다음이 2020년 연예·스포츠 기사 댓글을 폐지한 교훈(플랫폼 방조 책임·임시조치 의무·선수 초상권)을 반드시 반영해야 한다.

파일럿은 LG 트윈스, 두산 베어스 2팀만 오픈하며, 나머지 구단은 "곧 오픈" 상태로 노출만 한다. 라이트 모드 UI는 본 차수에서 제외하고 다크 모드만 구현한다.

## Goals / Non-Goals

**Goals:**
- 팀별 오픈 채팅방에서 텔레그램식 답장 스레드로 대화 가능
- 기사는 봇이 방에 핀 포스팅, 직접 댓글 기능은 제공하지 않음 (법적 리스크 회피)
- 메시지 단위 익명 토글 (가입은 실명 기반)
- 🔥 핫 기사 피드로 타팀 대화 가시화
- 신고·금칙어·AI 모더레이션·24시간 SLA까지 포함한 모더레이션 파이프라인
- iOS/Android 양 플랫폼 대칭 구현
- Supabase Auth(카카오·애플) 기반 인증

**Non-Goals:**
- 라이트 모드 UI
- 10개 구단 전체 오픈 (LG/두산만)
- 타팀 참여(메시지 작성) 정책 최종 결정 — 파일럿은 "읽기만" 옵션 A로 시작, 데이터 보고 후속 차수에서 재검토
- 기사 하위 직접 댓글 기능
- 웹 프로토타입의 커뮤니티 지원
- 워치(Wear OS/watchOS)의 커뮤니티 기능 지원
- 알림(push) 연동 — 후속 차수

## Decisions

### D1. 댓글 모델: 텔레그램식 답장 스레드 (기사 직접 댓글 X)

**결정**: 기사는 팀 채팅방의 핀 메시지로 포스팅하고, 사용자들은 메시지에 답장하는 방식으로 대화.

**대안**: 네이버식 "기사 하위 댓글 + 대댓글" 구조.

**이유**:
- 네이버·다음이 2020년 연예·스포츠 댓글을 폐지한 이유(선수 초상권·명예훼손 방조 책임)를 그대로 답습할 수 없다
- "기사에 달린 댓글"보다 "사용자 간 대화에 기사가 인용된 형태"가 법적 성격이 가볍다
- 텔레그램·디스코드 느낌이 야구 실시간 채팅 문화와 잘 맞는다
- 이미 앱에 WebSocket/Realtime 인프라가 있어 재활용 가능

### D2. 방 단위: 팀별 오픈 채팅방 1개 (기사별 방 X)

**결정**: 팀마다 메인 라운지 방 1개. 기사는 그 방에 핀 메시지로 들어간다.

**대안**: 기사마다 방 1개씩 자동 생성.

**이유**:
- 기사별 방은 대화가 파편화돼 활성화되지 않는다
- 야구 팬 문화는 "팀 단위로 같이 떠드는" 쪽에 가깝다
- 기사 상세 화면에서는 해당 기사에 달린 답장만 필터링해서 보여주는 "스레드 뷰"를 제공 (화면 3)

### D3. 인증: Supabase Auth + 카카오·애플 소셜 로그인

**결정**: Supabase Auth를 인증 백엔드로 사용. 공급자는 카카오·애플 2개만 지원.

**대안**:
- 자체 JWT 서버 구축 → 운영 부담 큼
- Firebase Auth → Supabase 생태계와 분리되어 이중 관리
- 구글·네이버 로그인 추가 → 범위 확장, 본 차수 Non-Goal

**이유**:
- Supabase는 이미 프로젝트 MCP에 연결돼 있고 RLS 기반 권한이 강력
- 카카오는 한국 야구 팬 커버리지 최대, 애플은 앱스토어 심사 요구사항
- 익명 로그인은 허용하지 않음 (법적 요청 대응을 위해 user_id 추적 필요)

### D4. 익명 토글: 메시지 단위 on/off, 방 단위 고정

**결정**: 메시지 작성 시 "익명" 토글을 켤 수 있고, 한 방 안에서는 해당 사용자의 익명 상태를 고정한다.

**이유**:
- 방 안에서 익명을 켰다 껐다 하면 다른 사용자들이 역추적 가능 (정체 노출)
- 내부 DB에는 `user_id`를 항상 저장 — UI 표시만 "익명"으로 가림
- 통신비밀보호법·정보통신망법상 법적 요청 대응 위해 실제 사용자 식별자 보관 의무

### D5. 핫 기사 랭킹: 가중치 공식 + 시간감쇠 + 5분 주기 뷰

**결정**: PostgreSQL materialized view를 5분마다 갱신.

**공식**:
```
raw_score = (고유 참여자 수 × 2) + (답장 수 × 1) + (좋아요 × 0.5) - (신고 수 × 3)
final_score = raw_score / POWER(경과 시간 hours + 2, 1.5)
```

**이유**:
- 단순 답장 수로 정렬하면 어그로·도배방이 1등 먹는다
- 신고 페널티 가중치(-3)가 핵심
- 파일럿 단계는 5분 주기 뷰로 충분, 트래픽 늘면 Redis sorted set으로 승격

### D6. 모더레이션 파이프라인: 3단계 방어

**결정**:
1. **금칙어 필터** (DB 테이블 기반) — 선수 실명 + 욕설 조합 자동 블라인드
2. **AI 모더레이션** — 메시지 작성 시 OpenAI Moderation API 또는 Google Perspective API 호출, 점수 임계치 초과 시 블라인드
3. **사용자 신고** — 신고 3회 누적 시 자동 블라인드, 24시간 내 운영자 검토

**이유**:
- AI 모더레이션만으로는 한국어 야구 맥락(선수 이름 + 혐오 표현) 커버 어려움
- 금칙어는 운영하면서 즉시 추가 가능
- 24시간 SLA는 정보통신망법상 임시조치 의무 대응

### D7. 기사 크롤러: 메타데이터만 저장, 본문 복제 금지

**결정**: 크롤러는 제목·요약(2~3줄)·썸네일 URL·원문 링크·출처·발행 시각만 저장. 본문 전체를 DB에 담지 않는다.

**이유**:
- 본문 복제는 저작권·DB권 침해 소지
- 썸네일은 재호스팅 없이 언론사 OG 이미지 URL만 참조
- 네이버 스포츠 공식 RSS/API 우선 사용, 없으면 robots.txt 준수 크롤링

### D8. 플랫폼 구현 전략: iOS·Android 대칭, 디자인 토큰 공유

**결정**: iOS(SwiftUI)·Android(Compose) 양 플랫폼에 동일 5화면. 기존 디자인 토큰 파일(`AppColors`, `AppSpacing`, `AppRadius`, `TeamTheme`) 확장해서 커뮤니티 전용 토큰 추가.

**이유**:
- 기존 앱이 iOS·Android 대칭이고 디자인 토큰이 레벨 0 동기화되고 있음
- Pencil 와이어프레임(`/pencil-new.pen`)이 이미 양 플랫폼 공통 기준
- 기사 크롤러·백엔드 API는 양 플랫폼이 공유하므로 모바일 계층만 플랫폼별 작업

## Risks / Trade-offs

- **[법적 방조 책임]** 선수 명예훼손·초상권 침해 메시지 방치 시 플랫폼 책임
  → **Mitigation**: 3단계 모더레이션 + 24시간 SLA + 이용약관 동의 + 금칙어 DB 초기값에 주요 선수명 등록

- **[저작권]** 기사 본문·썸네일 무단 복제
  → **Mitigation**: 본문 저장 금지, 썸네일은 URL 참조만, 원문 링크 노출 필수

- **[운영 부담]** 신고 24시간 SLA, 운영자가 검토해야 함
  → **Mitigation**: 자동 블라인드(신고 3회 or AI 점수 초과)로 1차 방어, 운영자 검토는 복구·제재 결정만. 파일럿 2팀 규모는 1인 운영 가능

- **[파일럿 활성도 실패]** 2팀 사용자 수가 적어 대화가 안 붙는 "유령 방" 문제
  → **Mitigation**: 봇이 기사 핀 포스팅으로 대화 씨앗 제공. 파일럿 기간 활성 지표 모니터링 후 팀 확장 결정

- **[인증 체계 신규 도입]** 기존 사용자가 로그인 없이 쓰던 앱에 로그인 강제
  → **Mitigation**: 커뮤니티 탭만 로그인 게이트, 홈·라이브·ME 탭은 기존처럼 비로그인 허용. "BREAKING"으로 명시

- **[타팀 어그로 방지 실패]** 타팀 참여를 제한해도 우회 가능
  → **Mitigation**: 파일럿 단계는 옵션 A(타팀 읽기만) 적용, 작성은 본인 응원팀 방만 허용. 데이터 수집 후 후속 차수에서 정책 결정

- **[Supabase Realtime 스케일]** 동시 접속 수가 많아지면 브로드캐스트 비용·지연 증가
  → **Mitigation**: 파일럿 2팀은 Realtime으로 충분. 확장 시 기존 Redis Pub/Sub 인프라로 이관 가능

## Migration Plan

1. **DB 스키마 추가** — 신규 테이블만 추가, 기존 테이블 변경 없음 (역호환)
2. **Supabase Auth 설정** — 카카오·애플 공급자 등록, 개발 환경부터 테스트
3. **백엔드 API 배포** — 신규 엔드포인트만 추가, 기존 API 영향 없음
4. **크롤러 배포** — Railway 크론, 초기엔 하루 N회로 시작
5. **모바일 앱 배포** — 커뮤니티 탭은 feature flag로 숨김 상태로 먼저 배포, 내부 테스트 후 오픈
6. **파일럿 오픈** — LG/두산만 방 생성, 타팀은 "곧 오픈" 표시
7. **롤백 전략** — feature flag off로 즉시 커뮤니티 탭 숨김. DB는 신규 테이블이라 롤백 불필요
