## Context

현재 백엔드 WebSocket은 경기 스트림 연결 직후 최신 상태와 최근 이벤트 목록을 초기 스냅샷으로 전송한다. 모바일 Android 서비스는 이 이벤트 목록을 라이브 이벤트와 동일하게 처리했고, 앱 재시작이나 스트림 재연결 시 과거 이벤트가 워치 햅틱 및 영상 알림으로 다시 전달될 수 있었다.

Wear OS Data Layer는 경로가 다른 DataItem을 별도 항목으로 유지한다. 기존 햅틱 전송은 이벤트마다 고유 경로를 사용했기 때문에 이전 이벤트 항목이 남을 수 있고, 워치가 재시작되거나 Data Layer가 재전달될 때 과거 항목이 다시 처리될 위험이 있었다.

플랫폼별 차이:
- 모바일 Android는 백엔드 WebSocket을 구독하고 워치로 경기 상태와 햅틱 이벤트를 전달한다.
- 모바일 iOS도 백엔드 WebSocket을 구독하고 WatchConnectivity로 Apple Watch에 경기 상태와 햅틱 이벤트를 전달한다.
- Wear OS는 Data Layer 수신 시 로컬 상태를 갱신하고 햅틱/이벤트 영상을 실행한다.
- watchOS는 WatchConnectivity 수신 시 로컬 상태를 갱신하고 햅틱/이벤트 영상을 실행한다.
- 웹은 워치 Data Layer와 무관하다.

배터리/네트워크 제약:
- 재연결 시 과거 이벤트를 재생하지 않으면 워치 화면 깨우기, 진동, 영상 재생을 줄여 배터리 소모를 낮출 수 있다.
- Data Layer 햅틱 경로를 최신 항목으로 유지하면 누적 항목 수와 재전달 처리량을 줄일 수 있다.

## Goals / Non-Goals

**Goals:**
- 모바일 Android 재연결 직후 초기 이벤트 스냅샷은 햅틱/영상 알림으로 전송하지 않는다.
- 모바일 iOS 재연결 직후 초기 이벤트 스냅샷은 햅틱/영상 알림으로 전송하지 않는다.
- 신규 실시간 이벤트는 기존처럼 워치 알림으로 전달한다.
- Wear OS가 stale 햅틱 이벤트를 수신해도 진동과 화면 깨우기, 이벤트 영상을 실행하지 않는다.
- 기존 누적 햅틱 경로 항목도 timestamp 기반으로 방어한다.

**Non-Goals:**
- 백엔드 WebSocket 초기 스냅샷 프로토콜을 변경하지 않는다.
- 경기 상태 UI의 최신성 보정 로직은 변경하지 않는다.
- watchOS 수신 stale 로직 추가 변경은 이번 변경에 포함하지 않는다. 기존 timestamp stale 방어를 확인한다.
- DB 스키마 또는 API 엔드포인트는 변경하지 않는다.

## Decisions

1. 모바일은 연결별 첫 이벤트 목록을 커서 기준선으로만 사용한다.
   - 이유: 백엔드가 연결 직후 보내는 이벤트 목록은 최근 이력 표시용 스냅샷이며, 사용자가 이미 지난 알림을 다시 받아서는 안 된다.
   - 대안: 백엔드 메시지에 초기 스냅샷 여부를 추가하는 방안이 있으나, 서버/클라이언트 프로토콜 변경 범위가 커서 이번 버그 수정에는 과하다.

2. 모바일 햅틱 Data Layer 경로는 최신 항목 경로로 갱신한다.
   - 이유: 이벤트마다 고유 경로를 만들면 Data Layer에 과거 이벤트가 누적될 수 있다. 최신 경로를 덮어쓰면 재시작 시 재전달 후보가 줄어든다.
   - 대안: 전송 후 DataItem을 삭제하는 방안이 있으나, 삭제 타이밍과 실패 처리까지 추가해야 해서 단순 최신 경로 갱신보다 복잡하다.

3. 햅틱 이벤트 payload에 발생 시각을 포함하고, 워치는 이를 stale 판단에 사용한다.
   - 이유: 워치가 이벤트 freshness를 독립적으로 판단해야 모바일/네트워크 재전달 상황에서도 안전하다.
   - 대안: 커서만으로 중복을 막는 방안은 기존 누적 경로 중 커서가 없던 이벤트를 방어하지 못한다.

4. 기존 누적 경로의 마지막 segment가 timestamp이면 워치가 fallback으로 사용한다.
   - 이유: 이미 설치된 앱 또는 기존 Data Layer 항목에 `updated_at`이 없는 경우에도 재시작 폭주를 막아야 한다.
   - 대안: 기존 항목 정리 작업을 별도로 두는 방안은 즉시 효과가 낮고 사용자 기기 상태에 따라 실패할 수 있다.

## Risks / Trade-offs

- [Risk] 연결 직후 발생한 실제 신규 이벤트가 초기 스냅샷 메시지에 섞여 있으면 알림이 생략될 수 있다. -> 백엔드가 연결 후 신규 변경은 별도 update로 브로드캐스트한다는 현재 동작을 전제로 하며, 상태 UI와 커서는 최신으로 유지한다.
- [Risk] 기기 시간이 크게 어긋나 있으면 stale 판단이 부정확할 수 있다. -> 현재 앱은 모바일과 워치의 일반적인 시스템 시간을 기준으로 동작하며, 10초 freshness 창으로 재전달 폭주를 우선 방어한다.
- [Risk] watchOS 직접 APNs payload에 발생 시각이 없으면 지연 push의 stale 판단이 제한된다. -> 이번 이슈의 재연결 초기 스냅샷은 iPhone 전송 단계에서 차단하고, WatchConnectivity 햅틱 메시지는 기존 timestamp stale 방어를 사용한다.

## Migration Plan

1. 모바일 Android, Wear OS, 모바일 iOS 앱을 함께 배포한다.
2. Android 모바일은 새 햅틱 경로와 timestamp를 사용한다.
3. Wear OS는 새 payload와 기존 timestamp 경로를 모두 stale 판단 대상으로 처리한다.
4. iOS 모바일은 연결별 초기 이벤트 스냅샷을 커서 기준선으로만 사용한다.
5. 문제가 발생하면 모바일의 초기 스냅샷 억제 변경과 Android 햅틱 경로 변경을 되돌리고, 워치 stale fallback은 유지해도 하위 호환에 영향을 주지 않는다.

## Open Questions
