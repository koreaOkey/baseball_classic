## Context

Wear OS 워치 앱은 모바일에서 전달받은 마지막 경기 상태를 로컬에 저장하고, 앱 화면과 타일이 이 값을 읽어 경기 화면을 표시한다. 종료된 경기 상태도 같은 저장소에 남기 때문에, 사용자가 경기 종료 후 앱을 다시 열면 최종 스코어를 확인할 수 있다.

문제는 다음날이 되어도 저장된 종료 경기 데이터가 남아 있다는 점이다. 화면 코드에는 자정 이후 종료 경기를 숨기는 분기가 있었지만 저장소 자체를 정리하지 않아 앱 화면과 타일의 정책이 중복되고, 이후 로직에서 오래된 캐시를 다시 참조할 여지가 있다.

## Goals / Non-Goals

**Goals:**
- 종료 당일에는 최종 경기 결과를 유지한다.
- 다음날 KST 00:00 이후에는 만료된 종료 경기 캐시를 삭제한다.
- 앱 화면과 타일이 동일한 만료 기준을 사용한다.
- 다음 경기 LIVE 감지 및 워치 동기화 흐름은 유지한다.

**Non-Goals:**
- 백그라운드 알람으로 정확히 자정에 앱을 깨워 캐시를 삭제하지 않는다.
- 예정 경기 전용 UI를 새로 만들지 않는다.
- 모바일 앱의 오늘 경기 캐시 정책을 변경하지 않는다.
- 백엔드 상태 전이 또는 DB 스키마를 변경하지 않는다.

## Decisions

1. **만료 시점: 다음날 KST 자정**
   - KBO 경기 날짜 기준과 일치하도록 Asia/Seoul 자정을 사용한다.
   - 종료 당일 밤에는 최종 스코어를 유지하고, 다음날부터는 경기 없음 상태로 전환한다.

2. **삭제 방식: 읽기 시점 lazy cleanup**
   - 앱 화면 또는 타일이 캐시를 읽을 때 만료 여부를 확인하고 삭제한다.
   - Wear OS 백그라운드 실행 제약과 배터리 소모를 피한다.

3. **공용 정책**
   - 앱 화면과 타일이 같은 만료/삭제 기준을 사용하도록 공용 helper를 둔다.
   - 경기 데이터와 마지막 이벤트 캐시만 삭제하고, 별도 동기화 프롬프트 키는 보존한다.

## Platform Notes

- **Android Watch:** 이번 변경의 직접 대상이다. 앱 화면과 타일 모두 영향을 받는다.
- **Android Mobile:** 워치로 경기 데이터를 보내는 기존 흐름은 유지한다.
- **iOS Mobile / watchOS:** 현재 변경 없음. watchOS 구현 시 동일한 “다음날 KST 자정 이후 종료 경기 만료” 정책을 맞출 수 있다.
- **DB / Backend:** 변경 없음.

## Risks / Trade-offs

- [사용자가 다음날에도 어제 최종 스코어를 보고 싶어 하는 경우] 워치에서는 경기 없음 UI가 우선된다. 모바일 홈의 과거 경기 결과 조회와 역할을 분리한다.
- [정확히 자정에 즉시 삭제되지 않음] 앱 화면 또는 타일이 다음에 읽는 시점에 삭제된다. 사용자에게 보이는 첫 렌더링 시점에는 이미 경기 없음 상태가 되므로 UX상 충분하다.
