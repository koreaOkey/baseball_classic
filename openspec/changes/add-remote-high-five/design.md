## Context

"야구봄"은 득점·홈런 순간을 워치 햅틱으로 전달하는 서비스다. 현재 이 경험은 개인 단말 안에서 끝난다. 본 change는 같은 시점에 **물리적으로 근처에 있는 같은 팀 팬**을 BLE로 감지하고, 양쪽 워치에 동시에 "하이파이브" 햅틱을 울려 "혼자 봐도 같이 본 것 같은" 순간을 만든다.

기존 환경 제약:
- 폰은 실시간 득점·홈런 이벤트를 WebSocket(`realtime` spec)으로 이미 수신 중이다. 새 이벤트 파이프라인은 필요 없다.
- 워치는 폰과 페어링된 상태에서 햅틱을 트리거한다(iOS WCSession, Android Wear Data Layer). 새 워치 ↔ 워치 통신은 없다.
- 커뮤니티 탭(`add-community-tab`)이 LG/두산 파일럿으로 준비 중이며, 같은 파일럿 플래그 아래에서 노출한다.
- 앱 이름은 사용자 노출 기준 "야구봄"이다.

## Goals / Non-Goals

**Goals:**
- 내 팀 득점·홈런 순간, 주변 같은 팀 팬이 있으면 양쪽 워치에 동시에 "하이파이브" 햅틱이 울린다.
- BLE만으로 동작한다(서버 위치 공유 없음). 위치 raw는 수집·저장·전송하지 않는다.
- 사용자는 설정 토글 하나로 on/off할 수 있고, 기본값은 off(명시적 opt-in)다.
- iOS/Android 양 플랫폼에서 같은 UX로 동작한다.
- 파일럿 팀(LG/두산)에서만 노출되며, 나머지 구단에서는 토글 자체가 보이지 않는다.
- 하이파이브 발생 로그가 커뮤니티 탭 피드에 **익명 집계 배지**("근처에서 N명과 하이파이브")로 남는다.

**Non-Goals:**
- 구장 지오펜스("직관 모드")는 별도 change(`add-stadium-mode`)로 분리.
- 서버 위치 공유, GPS 기반 근접 감지, 친구 관계 기반 하이파이브.
- 하이파이브 "상대"의 프로필·닉네임·아이디 노출. 하이파이브는 **익명**이다.
- 웹 프로토타입(Real-time Baseball Broadcast) 반영.
- 실시간 채팅이나 메시지. 본 change는 햅틱 한 방과 배지 한 줄이 전부다.

## Decisions

### 1. 근접 감지 방식: BLE 광고(P2P) 채택, 서버 위치 공유 거절
- **채택**: BLE 광고 + 스캔. 폰이 이벤트 수신 시점에 짧게(≤20초) 광고를 켜고, 동시에 주변을 스캔한다.
- **거절안**: (a) GPS 좌표를 서버에 올리고 반경 쿼리, (b) Apple Nearby Interaction / Google Nearby Share.
- **이유**: 위치 raw를 서버로 보내지 않는다는 프라이버시 원칙, "근처(수 m~수십 m)"라는 의미에 BLE RSSI가 자연스럽다, 구장·카페 같은 실내에서도 동작, 크로스플랫폼(iOS↔Android) 가능.
- **트레이드오프**: iOS 백그라운드 BLE 광고 제약(서비스 UUID만 광고 가능, local name 불가), Android 12+ 권한 분리 — 뒤 Risks 참조.

### 2. 트리거 윈도우: 득점·홈런 이벤트 수신 시점에만 open
- 내 팀 득점·홈런 이벤트가 들어온 순간 ±20초 동안만 BLE 광고·스캔 창이 열린다. 평시 상시 스캔은 하지 않는다.
- **이유**: 배터리와 프라이버시 양쪽에서 최소 권한 원칙. 사용자는 "사건 있을 때만 근처를 본다"는 모델을 이해하기 쉽다.
- 같은 경기에서 여러 번 득점해도 각 이벤트마다 새 윈도우가 열린다. 윈도우 중복이 있으면 기존 윈도우를 연장.

### 3. 광고 페이로드: 익명 팀 ID + 회전 토큰
- Service UUID는 야구봄 전용 1개 고정(하이파이브 프로토콜 식별용).
- Service Data 16바이트 이내에 다음을 싣는다:
  - `team_id` (1 byte): KBO 팀 식별자
  - `event_type` (1 byte): 1=득점, 2=홈런
  - `rotating_token` (8 bytes): 1분마다 로컬에서 회전하는 랜덤 토큰. 사용자·기기와 연결 가능한 식별자가 아니다.
  - `protocol_version` (1 byte)
- 닉네임·UUID·계정ID·폰번호 일체 금지.
- **이유**: 추적 불가능한 프라이버시, MAC 랜덤화와 함께 동일 사용자의 시계열 추적을 차단.

### 4. 매칭 규칙: 같은 팀 + 같은 이벤트 종류 + RSSI 임계값
- 수신 광고 중 `team_id == 내 팀` 이고 `event_type == 내가 방금 받은 이벤트`이며 `RSSI ≥ -80 dBm`(실내 수십 m)인 경우에만 "하이파이브 성립"으로 판정.
- **이유**: 다른 팀 팬 혹은 먼 거리 우연 매칭을 제외하고, "같이 보고 있구나"가 성립하는 상황만 남긴다.
- RSSI 임계값은 플래그로 관리하여 파일럿 중 튜닝 가능.

### 5. 햅틱: 기존 득점 패턴과 구분되는 전용 패턴
- "하이파이브 성립"이 판정된 경우, 기존 득점 햅틱 직후 **전용 "하이파이브" 패턴**(예: 짧은 이중 탭 + 긴 펄스)을 덧붙인다.
- 기존 득점 햅틱 자체는 건드리지 않는다(햅틱 정규화 규칙 유지).
- 워치 전용 패턴은 iOS(`WKHapticType`/커스텀 시퀀스) / Wear OS(`VibrationEffect`) 각 플랫폼 고유 API로 구현.

### 6. 커뮤니티 탭 피드: 집계 배지 한 줄
- 하이파이브 성립 시, 폰은 커뮤니티 탭 API에 "이 경기 이 이벤트에서 N명과 하이파이브" 아이템을 기록한다(파일럿 팀 한정).
- 개별 하이파이브 **상대의 식별자는 저장하지 않는다**. 저장되는 것은 경기 ID, 이벤트 시점, 내 팀, 본인의 하이파이브 카운트뿐이다.
- 피드 아이템 타입 1종 추가. 커뮤니티 API 스키마는 `add-community-tab`과 협의.

### 7. 권한·opt-in 정책
- 설정 토글 **기본 off**. 토글을 켤 때 비로소 BLE 권한을 요청한다.
- iOS: `NSBluetoothAlwaysUsageDescription` 문구에 "근처의 같은 팀 팬을 감지해 하이파이브 햅틱을 보내기 위해 사용합니다. 위치 정보는 수집하지 않습니다."
- Android 12+: `BLUETOOTH_SCAN` + `BLUETOOTH_ADVERTISE`에 `neverForLocation` 플래그. 위치 권한은 요청하지 않는다. Android 11 이하는 파일럿 지원 범위에서 제외.
- 토글 off 전환 시 즉시 광고·스캔 중단 및 권한 사용 금지.

### 8. 플랫폼별 역할
- **iOS mobile**: CoreBluetooth central+peripheral. `realtime` 이벤트 훅에서 윈도우 open. 워치 햅틱은 WCSession으로 트리거.
- **Android mobile**: `BluetoothLeAdvertiser` + `BluetoothLeScanner`. `realtime` 이벤트 훅에서 윈도우 open. Wear Data Layer로 워치 햅틱 트리거.
- **watchOS / Wear OS**: 폰에서 받은 "하이파이브 트리거" 메시지에 대해 전용 햅틱 패턴 재생만 담당. BLE는 사용하지 않는다(배터리).
- **Backend**: 본 change에서는 변경 없음(커뮤니티 피드 아이템 타입 추가만).

## Risks / Trade-offs

- **[iOS 백그라운드 BLE 제약]** 앱이 백그라운드·잠금 상태일 때 광고는 local name 없이 service UUID만 노출되고, 스캔도 필터 기반으로 제약된다. → 본 기능은 "득점 이벤트를 방금 받은 시점"에만 활성이고, 그 순간 앱은 이미 백그라운드에서 깨어 있으므로 허용 범위 안. 백그라운드에서 이벤트 수신이 없는 경우는 기능 대상이 아니다.
- **[Android 12 이전 디바이스]** `BLUETOOTH_SCAN`/`ADVERTISE` + `neverForLocation` 정책이 성립하지 않아 위치 권한을 요구하게 된다. → 파일럿 지원 범위에서 Android 11 이하 제외. 토글이 보이지 않게 한다.
- **[배터리]** 상시 스캔은 치명적. → 이벤트 트리거형 윈도우(≤20초)로 한정해 이 경로를 원천 차단.
- **[MAC 랜덤화와 매칭 실패]** OS가 MAC을 랜덤화해도 본 기능은 "같은 기기 재식별"이 필요 없다(성립 여부만 판정). → 설계상 문제 없음.
- **[스푸핑/악의적 광고]** 누군가 수많은 가짜 광고를 뿌려 N 카운트를 부풀리기. → RSSI 임계값, 윈도우 시간 제약, 동일 `rotating_token` 중복 집계 방지. 파일럿 단계에서 오남용 로그 모니터링.
- **[크로스 플랫폼 호환성]** iOS↔Android BLE 광고 수신은 일반적으로 가능하지만, Service Data 파싱 차이가 있다. → 초기 통합 테스트 필수 항목으로 tasks.md에 포함.
- **[프라이버시 오해]** "위치를 공유한다"고 오해할 수 있다. → 토글 설명, 권한 요청 다이얼로그, 개인정보 처리방침 문구에 "위치 raw를 수집·저장·전송하지 않음"을 명시.
- **[파일럿 플래그 누수]** 비파일럿 팀 사용자가 토글을 보게 되면 기대를 실망시킨다. → 팀별 기능 플래그를 서버/빌드 상수 둘 다에 넣어 이중 게이트.
- **[비활성 경기 시간 오작동]** 이벤트 훅이 과거 이벤트를 지연 수신해 윈도우를 여는 경우. → 이벤트 타임스탬프와 현재 시각 차이가 60초 이상이면 윈도우를 열지 않는다.

## Migration Plan

- **전제**: 본 change는 신규 기능이며 제거할 기존 동작이 없다.
- **롤아웃 단계**
  1. 내부 드로그푸드: 사내 기기 몇 대에서 토글 수동 on. 득점 이벤트 시뮬레이터로 윈도우·매칭·햅틱 검증.
  2. 파일럿 릴리즈: LG/두산 플래그 on 상태로 스토어 업데이트. 커뮤니티 탭 배지 누적 지표 + 오남용 로그 모니터링.
  3. 일반 릴리즈: 파일럿 지표 양호 시 전 구단 플래그 on.
- **롤백**: 서버 플래그 1개로 즉시 토글 숨김 및 윈도우 open 차단. 기존 득점 햅틱 경로는 영향 없음.
- **개인정보 처리방침** 문구 업데이트가 선행되어야 스토어 심사 리스크를 줄인다.

## Open Questions

- 커뮤니티 탭 피드 아이템 타입을 새로 만들지, 기존 "경기 이벤트 카드"에 카운트 필드를 붙일지 — `add-community-tab` 오너와 조율 필요.
- 하이파이브 알림 문구 카피("○○ 팬 N명과 하이파이브!") 최종 톤.
- 득점·홈런 외에 "경기 시작", "끝내기" 같은 이벤트도 포함할지 — 파일럿 후 검토.
- RSSI 임계값(-80 dBm)의 실제 실내·실외 튜닝 결과.
- 동일 경기 같은 이벤트 윈도우 내 중복 `rotating_token` 처리(집계 dedupe)의 정확한 기준.
