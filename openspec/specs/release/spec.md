# Release Specification

## Purpose
BaseHaptic(야구봄)의 브랜치 전략, 태그 컨벤션, 배포 체크리스트, 롤백 절차, 릴리즈 이력 관리를 정의한다. iOS·Android·backend·crawler 4개 플랫폼을 모노레포에서 운영하면서 배포 시점과 히스토리 일관성을 유지하기 위한 단일 기준 문서다.

## Requirements

### Requirement: 브랜치 전략
프로젝트는 `main` 브랜치를 항상 배포 가능한 상태로 유지하며, 모든 작업은 수명이 짧은 `feature/*` 브랜치에서 수행된 뒤 `main`에 머지되고 삭제되어야 한다. 긴급 수정은 해당 릴리즈 태그에서 파생된 `hotfix/*` 브랜치에서 수행한다.

#### Scenario: main 브랜치는 항상 배포 가능
- **WHEN** `main` 브랜치의 임의 커밋에서 빌드를 수행한다
- **THEN** 해당 빌드는 배포 가능한 상태여야 하며, 빌드 실패 또는 반쯤 완성된 기능이 포함되어 있지 않아야 한다

#### Scenario: 기능 작업은 feature 브랜치에서 수행
- **WHEN** 새로운 기능·리팩토링·버그 수정을 시작한다
- **THEN** `feature/<작업명>` 브랜치를 `main`에서 분기하여 작업한 뒤, 머지 완료 후 해당 브랜치를 삭제한다

#### Scenario: 핫픽스는 릴리즈 태그에서 분기
- **WHEN** 이미 배포된 버전에서 긴급 수정이 필요하다
- **THEN** 해당 릴리즈 태그에서 `hotfix/<플랫폼>-<버전>` 브랜치를 분기하여 수정한 뒤, 새 패치 태그를 생성한다

### Requirement: 태그 컨벤션
릴리즈 시점은 git 태그로 기록되어야 하며, 태그는 통합 릴리즈와 플랫폼별 릴리즈 두 가지 형식을 허용한다.

#### Scenario: 통합 릴리즈 태그
- **WHEN** iOS·Android·backend·crawler가 동일 시점에 함께 배포된다
- **THEN** 해당 커밋에 `v{major}.{minor}.{patch}` 형식의 태그(예: `v1.0.0`)를 부착하고 원격에 푸시한다

#### Scenario: 플랫폼별 릴리즈 태그
- **WHEN** 특정 플랫폼만 독립적으로 배포된다
- **THEN** 해당 커밋에 `{platform}-v{major}.{minor}.{patch}` 형식의 태그를 부착한다. 허용되는 platform 값은 `ios`, `android`, `backend`, `crawler` 네 가지이다

#### Scenario: 태그 메시지에 배포 범위 명시
- **WHEN** 릴리즈 태그를 생성한다
- **THEN** 태그 메시지에 어떤 플랫폼이 포함되었는지와 주요 변경 요약을 1~2줄로 기록한다

### Requirement: 배포 체크리스트
배포는 "태그가 존재하고 원격에 푸시되었으며 릴리즈 이력에 기록되었을 때" 완료된 것으로 본다.

#### Scenario: 배포 완료 정의
- **WHEN** 어떤 플랫폼의 배포가 "완료"되었다고 선언된다
- **THEN** 다음 세 조건이 모두 충족되어야 한다: (1) 해당 시점 커밋에 규약에 맞는 태그가 부착되어 있다, (2) 태그가 원격 저장소에 푸시되어 있다, (3) 릴리즈 이력 섹션에 해당 항목이 추가되어 있다

#### Scenario: 태그 누락 감지
- **WHEN** 배포 직후 릴리즈 이력을 갱신하려 한다
- **THEN** 태그가 부착되지 않은 경우 배포가 완료되지 않은 것으로 간주하고, 즉시 태그를 생성하고 원격에 푸시한다

### Requirement: 롤백 절차
과거 배포 상태로 복귀가 필요할 때는 파괴적 연산(예: `git reset --hard` + force push) 대신 태그 기반 체크아웃과 핫픽스 브랜치를 사용해야 한다.

#### Scenario: 과거 배포 상태 확인
- **WHEN** 특정 릴리즈 시점의 코드를 확인하려 한다
- **THEN** 해당 태그를 `git checkout <tag>`로 체크아웃하여 상태를 확인한다

#### Scenario: 과거 배포로부터 수정 후 재배포
- **WHEN** 과거 릴리즈에서 긴급 수정이 필요하다
- **THEN** 해당 태그에서 `hotfix/<플랫폼>-<버전>` 브랜치를 분기해 수정한 뒤, 새 패치 버전 태그를 생성하여 배포한다

#### Scenario: main 브랜치 강제 되돌림 금지
- **WHEN** 과거 상태로 돌리고 싶은 유혹이 있다
- **THEN** `git reset --hard` + force push로 `main`을 과거로 이동시키는 방식은 금지되며, 대신 `git revert`로 새 커밋을 만들거나 핫픽스 브랜치를 사용한다

### Requirement: 릴리즈 이력 관리
모든 릴리즈는 버전, 커밋 SHA, 플랫폼, 날짜, 비고를 포함하는 이력 표로 spec 내부에 유지되어야 한다.

#### Scenario: 신규 릴리즈 기록
- **WHEN** 새 릴리즈 태그가 생성된다
- **THEN** 릴리즈 이력 섹션에 해당 버전·커밋 SHA·플랫폼·날짜·비고가 추가된다

#### Scenario: 잘못된 태그 교정 기록
- **WHEN** 과거 태그가 잘못된 커밋을 가리키고 있음을 발견해 교정한다
- **THEN** 교정 내역(이전 SHA → 새 SHA, 교정 사유)을 이력 섹션의 비고에 명시한다

### Requirement: 단독 운영 예외 조항
이 프로젝트는 현재 단독 운영 상태이므로, 일반적으로는 금지되는 태그 force-update가 한시적으로 허용된다. 이 예외는 협업자가 합류하는 시점에 즉시 재검토되어야 한다.

#### Scenario: 잘못된 태그의 교정
- **WHEN** 원격에 이미 푸시된 태그가 잘못된 커밋을 가리키고 있음이 확인된다
- **THEN** 로컬에서 태그 삭제, 원격 태그 삭제, 올바른 커밋에 재생성, 원격 재푸시 순으로 교정할 수 있다

#### Scenario: 협업자 합류 시 재검토
- **WHEN** 이 레포에 본인 외 다른 커밋 권한자가 추가된다
- **THEN** 태그 force-update 예외 조항은 즉시 무효화되며, 일반적인 불변 태그 규칙으로 전환되어야 한다
