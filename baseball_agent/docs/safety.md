# Safety · 저작권 가드 규칙

> 본 Agent는 KBO 선수·구단을 모티프로 삼는 이미지를 생성한다. **초상권·퍼블리시티권·상표권**이 직결되므로 이 문서에 명시된 규칙을 Agent 전체 Workflow에 강제한다. 규칙 위반 이미지는 저장·반환되지 않는다.

---

## 1. 핵심 원칙

1. **실제 인물의 동일성은 재현 금지.** 실존 선수 얼굴·초상·신체 특징은 어떤 스타일로도 그리지 않는다.
2. **구단 공식 로고·마스코트 IP 재현 금지.** 삼성 블레오·두산 곰돌이 등 공식 마스코트의 고유 디자인은 그대로 따라 쓰지 않는다.
3. **일반 동물 마스코트는 허용.** 픽사풍 아기 동물(사자·호랑이·곰·독수리 등)을 "야구 유니폼 입은 귀여운 캐릭터"로 그리는 것은 OK. 단 팀 공식 마스코트의 컬러·의상·포즈를 그대로 모사하면 안 된다.
4. **사실 정보 활용은 허용.** 등번호(생략 또는 일반 번호)·팀 공식 컬러·포지션·플레이 모션은 사실이므로 사용 가능.
5. **실패는 SKIP이다.** 가드를 통과 못한 이미지는 재시도 3회 후 폐기하고 해당 슬롯을 비운 채 팩을 마무리한다.
6. **법적 확신이 없으면 더 일반적으로 간다.** 경계 케이스에서는 항상 안전한 방향으로 후퇴.

---

## 2. 금지·허용 매트릭스

### ❌ 금지 (MUST NOT)

| 범주 | 금지 사항 | 이유 |
|---|---|---|
| 인간 얼굴 | 실존 인물의 얼굴·눈·코·입 구체 묘사 (사실화·일러스트 모두) | 초상권 |
| 인간 인물 | 식별 가능한 실제 선수 신체 특징 (헤어·수염·타투) | 퍼블리시티권 |
| 구단 로고 | KBO·각 구단 공식 로고·엠블럼 재현 | 상표권 |
| 공식 마스코트 | 블레오·호범이 등 각 구단 공식 마스코트의 고유 디자인 모사 | 상표·저작권 |
| 유니폼 엠블럼 | 실제 유니폼 등판·가슴 엠블럼·스폰서 로고 | 상표·계약 |
| 합성 캐릭터 | 다른 유명 IP 캐릭터(미키·피카츄 등) 차용·혼합 | 저작권 |
| 텍스트 | "MLB", "KBO", 구단명 영문 로고체, 선수 이름 | 상표 |

### ✅ 허용 (사실 모티프 + 일반 마스코트)

| 범주 | 허용 사항 | 근거 |
|---|---|---|
| 번호 | 일반 등번호 표기 (선수 실번호 아니어도 무방) | 일반 요소 |
| 컬러 | 팀 공식 컬러 코드 (예: 두산 네이비 #0A2351) | 사실 정보 |
| 포지션 | 투수·포수·CF 등 포지션 실루엣·아이콘 | 일반 야구 도상 |
| 모션 | 배팅 스윙·슬라이딩·투구·수비 제스처 | 공개 스포츠 도상 |
| 장비 | 배트·글러브·헬멧·스파이크 (일반 디자인) | 일반 도구 |
| 배경 | 야구장의 추상적 실루엣, 잔디·흙·조명 | 공개 공간 |
| 계절/기후 | 벚꽃·야경·조명 등 무드 | 일반 풍경 |
| **일반 동물 마스코트** | 아기 사자·호랑이·곰·독수리·공룡 등, 픽사풍 3D 또는 플랫 일러스트 | 일반 창작 캐릭터 |
| **그래픽 요소** | 야구공·하트·선버스트·기하 패턴 | 일반 도상 |
| 별명 | 선수 공식 별명 (예: "바람의 손자") | 별명 자체는 OK, 시각화는 일반 모티프로 |

**동물 마스코트 가이드**: 팀 이름에서 파생된 동물(라이온즈=사자, 타이거즈=호랑이 등)은 허용. 단 아래 준수:
- 공식 마스코트(예: 블레오)의 얼굴·의상·포즈를 그대로 모사하지 않음
- 일반적인 "귀여운 아기 {동물}" 스타일로 그림
- 유니폼은 플레인 무지(팀 색상만 반영, 로고·엠블럼 없음)

## 3. 번역 규칙 — 입력 문구를 안전한 프롬프트로

`EXTRACT_TRAITS` Node는 Research 결과에서 **금지 어휘를 제거하고 허용 모티프로 치환**해야 한다.

| 입력 문구 | ❌ 그대로 쓰면 위험 | ✅ 안전한 치환 |
|---|---|---|
| "이정후의 얼굴" | 초상권 | "11번 배번·샌프 오렌지·빠른 스윙 실루엣" |
| "두산 로고 넣어줘" | 상표권 | "두산 공식 네이비·B 형태 추상 기호" |
| "김혜성과 꼭 닮게" | 초상권 | "2루수 포지션 실루엣·빠른 발 모션·등번호" |
| "유니폼 그대로" | 상표·계약 | "팀 컬러 기반 추상 스트라이프 패턴" |

### 프롬프트 템플릿 필수 포함 문구 (DALL-E 프롬프트 끝에 항상 붙인다)

```
Style: abstract pictogram, silhouette composition, no human face,
no real person, no official team logo, no text, no uniform emblem,
geometric shapes only.
```

## 4. 검증 단계 — `face_logo_detector` Tool

모든 생성 이미지는 저장 직전에 `face_logo_detector` Tool(GPT-4o Vision)을 거친다.

### 판정 기준

| 감지 대상 | 판정 | 조치 |
|---|---|---|
| 사람 얼굴 (눈·코·입 식별 가능) | FAIL | 추상화 레벨 +1 → 재생성 |
| 사람 얼굴 (실루엣만) | PASS | 저장 |
| 구단 공식 로고 (동일성 80%+) | FAIL | 프롬프트에서 로고 키워드 제거 → 재생성 |
| 팀 컬러 조합만 사용 | PASS | 저장 |
| 다른 유명인·캐릭터 | FAIL | 전면 폐기 → SKIP |
| 부적절 콘텐츠 (폭력·성적) | FAIL | 전면 폐기 → SKIP |

### Tool 반환 스키마

```json
{
  "has_face": true | false,
  "has_logo": true | false,
  "logo_similarity": 0.0,
  "has_other_ip": true | false,
  "has_inappropriate": true | false,
  "reason": "설명 문자열",
  "verdict": "pass" | "refine" | "skip"
}
```

## 5. 실패 처리 정책

| 실패 유형 | 재시도 횟수 | 후속 조치 |
|---|---|---|
| 얼굴 감지 | 3회 | 추상화 레벨 상향 (프롬프트에 `pictogram only` 강화) |
| 로고 감지 | 3회 | 프롬프트에서 구단명·로고 키워드 제거 |
| 대비 실패 (WCAG) | 3회 | 전경·배경 대비 지시 강화 |
| 다른 유명인·부적절 | **0회** | 즉시 SKIP (재시도 없음) |
| API 에러 | 2회 | 지수 백오프 후 재시도 |

3회 실패 시 해당 슬롯(이벤트·테마 요소)은 `status: "failed_after_retry"`로 manifest에 기록하고 팩 저장은 계속 진행(graceful degradation).

---

## 6. 데이터 수집 단계 가드

`web_search` Tool로 선수 정보를 조회할 때도 다음 원칙을 지킨다.

- 검색 쿼리에 선수 **실명**은 쓰되, 반환된 이미지·얼굴 묘사는 **다음 단계로 넘기지 않는다**.
- 웹 결과 요약 시 LLM이 "얼굴이 둥글고", "머리가 검고" 같은 외모 묘사를 뽑았다면 `EXTRACT_TRAITS` Node에서 **드롭**한다.
- 허용된 것은 **팩트**(포지션·기록·팀·최근 이슈 요약문)뿐이다.

---

## 7. 팩 산출물에 대한 메타 가드

`manifest.json`과 이벤트별 `meta.json`에 선수 실명은 들어가도 괜찮다(파일 내부 메타, 일반 공개 배포 아님). 단:

- 파일명에는 **선수 실명 영문·한글 풀네임 사용 지양**, 대신 `pack_20260423_01_leejungho` 처럼 가벼운 슬러그로.
- 이미지 파일 자체에는 선수 이름 텍스트를 렌더링하지 않음(프롬프트에 `no text` 지시).

## 8. 개발자 체크리스트

기능 구현 시 다음 항목을 반드시 확인한다.

- [ ] DALL-E 프롬프트에 **안전 템플릿 문구** 4줄이 자동 삽입되는가
- [ ] `EXTRACT_TRAITS` Node 출력에 금지 어휘가 남아있지 않은가 (얼굴·초상 관련)
- [ ] 모든 생성 이미지가 `face_logo_detector`를 **반드시 경유**하는가 (bypass 불가)
- [ ] 재시도 상한 3회가 강제되는가 (무한 루프 방지)
- [ ] "다른 유명인·부적절" 판정 시 **즉시 SKIP** 경로가 있는가
- [ ] `manifest.json`에 실패 슬롯이 `status: "failed_after_retry"`로 기록되는가
- [ ] 프롬프트·결과·비용이 `stdout` JSON progress event로 투명하게 공개되는가

## 9. 한계와 후속 과제

- 본 가드는 **MVP 수준**이다. 실제 야구봄 앱 배포 전에는 외부 법무 검토를 별도 openspec change로 진행한다.
- 현재 `face_logo_detector`는 GPT-4o Vision 기반이라 100% 정확하지 않다. 중대 결과 이미지는 사람 최종 검수가 필요하다.
- KBO 구단 로고·유니폼의 "동일성" 판정은 주관적 요소가 있다. 경계 케이스는 보수적으로(FAIL) 판정한다.
- 과제 발표용 데모는 사전에 사람 검수를 거친 안전 이미지만 시연한다.
